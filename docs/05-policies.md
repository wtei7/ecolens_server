# 5. 정책 (위치 기록 · AI 처리 · 예외 처리)

## 5.1 위치 기록 정책

### 위치 저장 조건

다음 조건 중 하나를 만족하면 새로운 위치를 저장한다.

- 이전 저장 시점에서 **5초 이상** 경과
- 이전 저장 위치에서 **5m 이상** 이동

### 위치 데이터 제외 조건

다음 중 하나라도 해당하면 저장하지 않는다.

- 위치 정확도가 **50m 이상**
- 직전 위치에서 비정상적으로 멀리 이동 (점프)
- 이동 속도가 현실적인 보행 속도를 크게 초과
- 위도 또는 경도 값이 유효하지 않음 (범위·NaN 등)

### 서버 측 처리

- Batch API 수신 시 각 좌표에 대해 위 필터를 적용.
- 필터링 결과 `accepted` / `rejected` 카운트를 응답.
- `sequence`는 서버에서 단조 증가 보장.
- PostGIS `ST_DistanceSphere` 등으로 거리/속도 검증 가능.

```sql
-- 예: 5m 미만 이동 + 5초 미만 경과 → 스킵
-- 예: 정확도 >= 50m → 스킵
```

### 네트워크 연결 실패

- 모바일 앱은 위치 데이터를 내부 저장소에 임시 저장.
- 네트워크 복구 시 저장된 좌표를 **Batch API**로 전송.
- 서버는 순서(`sequence`)와 시간(`recorded_at`)을 기준으로 정렬 저장.
- 서버는 중복 Batch(재전송)를 멱등하게 처리하도록 권장(예: 좌표 식별용 클라이언트 키 또는 idempotency-key).

## 5.2 GPS 튐 현상 처리

- 비정상 위치는 필터링하고 지도 경로에 반영하지 않음.
- 점프 판정 로직 예:
  - 이전 좌표와의 거리 / 시간 차 = 속도
  - 보행 속도 임계치(예: > 7 m/s) 초과 시 스킵
- 필터링된 데이터는 별도 로그로 기록(추후 디버깅/지표용)하되, 경로에는 포함하지 않는다.

## 5.3 중복 탐지 방지 정책

같은 쓰레기가 여러 프레임에서 반복 탐지될 수 있으므로, 서버에서 다음 조건을 모두 만족하면 **동일한 쓰레기**로 판정한다.

| 조건 | 임계값 |
| --- | --- |
| 같은 플로깅 세션 | `session_id` 일치 |
| 같은 쓰레기 종류 | `type` 일치 |
| 탐지 시간 차이 | ≤ 15초 |
| 기존 기록과 거리 | ≤ 8m |

동일한 쓰레기로 판정되면:

- 새 레코드를 생성하지 않는다.
- 기존 레코드의 `detection_count`를 +1.
- `last_detected_at`을 최신 탐지 시간으로 갱신.
- `detection_logs`에는 새 탐지 로그를 추가(bounding box, confidence, image).

### 거리 계산(PostGIS)

```sql
ST_DWithin(
  location,
  ST_SetSRID(ST_MakePoint(:lon, :lat), 4326),
  8   -- meters
)
```

또는 `ST_DistanceSphere`를 사용해 meter 단위 거리 비교.

## 5.4 쓰레기 위치 자동 등록 정책

- 탐지 신뢰도가 **0.7 이상**인 detected 결과만 쓰레기 기록으로 저장.
- 현재 GPS 좌표와 함께 저장.
- 중복 검사 수행 후 신규 생성 여부 결정.
- 저장 정보는 [03-database.md](./03-database.md)의 `trash_records` 참고.

```text
쓰레기 탐지
→ 신뢰도 >= 0.7 확인
→ 현재 GPS 위치 확인
→ 중복 데이터 검사
→ 쓰레기 기록 생성 (또는 기존 레코드 갱신)
→ detection_logs 추가
```

## 5.5 정화 검증 정책

### 기본 흐름

```text
쓰레기 탐지(detected)
→ 사용자가 수거
→ 사용자가 After 이미지 촬영
→ AI 서버에 Before·After 비교 요청 (POST /api/ai/verify-cleaning)
→ 결과: cleaned | not_cleaned | uncertain
→ 상태 갱신 + 점수 지급(cleaned만)
```

### 결과별 서버 동작

| 결과 | 상태 전환 | 점수 | 비고 |
| --- | --- | --- | --- |
| `cleaned` | `cleaned` | 지급 (점수 테이블 기반) | 누적 점수/정화 수 갱신 |
| `not_cleaned` | `not_cleaned` | 미지급 | 미수거로 유지 |
| `uncertain` | `uncertain` 또는 `verification_pending` | 미지급 | 재촬영 요청 또는 수동 확인 |

- 점수는 세션 종료 시 합산하여 `total_score`에 반영.
- 사용자 `total_score`, `total_cleaned_count`도 함께 갱신.

## 5.6 AI 처리 정책

### 탐지 요청 최적화

- 전체 영상을 저장하지 않는다.
- 1~2초 간격으로 프레임 추출 (클라이언트 측).
- 서버 전송 전 이미지 압축 (클라이언트 측).
- 동일 위치에서 과도한 요청이 발생하지 않도록 서버 측 rate-limit 권장.
- 앱이 백그라운드이면 분석 중지 (클라이언트 측).

### 탐지 실패 처리

- AI 서버가 응답하지 않더라도 **GPS 경로 기록은 계속 저장**되어야 한다.
- AI 탐지와 플로깅 세션 기능을 분리하여 한 기능의 장애가 전체 활동을 중단시키지 않도록 한다.
- 클라이언트에 탐지 실패 메시지를 반복 노출하지 않는다.
- 일정 시간 후 자동 재시도(지수 백오프 권장).

### Mock AI 모드

```env
AI_MODE=mock   # mock | yolo
```

- Mock 모드에서는 고정/랜덤 테스트용 탐지 결과를 반환.
- 백엔드는 `AI_MODE`에 따라 `detect_service` / `verify_service`가 mock/YOLO 구현을 선택하도록 설계(전략 패턴).
- Mock 응답 예시는 [04-api.md](./04-api.md)의 `POST /api/ai/detect` 응답 참고.

## 5.7 예외 처리 정책

### 위치 권한 거부 (클라이언트 → 서버 영향)

- 플로깅 시작 불가 → `/api/plogging/sessions` 생성 자체를 클라이언트가 막음.
- 서버는 유효한 세션 요청만 수신하므로 추가 검증 불필요. 단, 좌표 없이 세션이 생성되는 경우를 대비해 위도/경도를 optional로 수용.

### 카메라 권한 거부

- AI 탐지 없이 경로 기록만 진행.
- 서버는 `/api/ai/detect` 호출이 없어도 정상 동작해야 함.

### 네트워크 연결 끊김

- 위치/탐지 데이터는 모바일 로컬에 임시 저장 후 복구 시 Batch 동기화.
- 서버는 Batch API를 멱등 처리(중복 수신 방지).
- 시간 역전/과거 좌표 보정을 허용하되 `recorded_at` 기준 정렬.

### 앱 강제 종료

- 활성 세션(`active`)은 서버에 그대로 남아 있음.
- 앱 재실행 시 `GET /api/plogging/sessions?status=active`로 복구 여부 확인.
- 서버는 일정 시간(예: 24h) 이상 활성인 세션을 자동 만료/`cancelled` 처리하는 배치 권장.

### AI 서버 오류

- 사용자 활동은 유지 (세션, 경로는 정상 저장).
- 탐지 실패 응답(502/504 타임아웃)은 클라이언트가 억제 노출.
- 서버는 AI 호출 실패 시 502를 반환하고, 상세 에러는 로그에만 기록.

### GPS 튐 현상

- 비정상 위치 필터링, 경로에 반영하지 않음.
- 서버 Batch 처리에서 동일하게 필터링 적용.

## 5.8 점수 계산 정책 요약

| 쓰레기 종류 | 점수 |
| --- | ---: |
| 일반 쓰레기 | 3 |
| 종이컵 | 4 |
| 캔 | 5 |
| 페트병 | 6 |
| 비닐 | 6 |
| 유리병 | 8 |
| 대형 쓰레기 | 20 |

- MVP는 콤보 배율 미적용. 기본 점수만 계산.
- 정화 완료(`cleaned`)된 쓰레기에만 점수 지급.
- 총점 = 정화 완료 쓰레기 점수 합계.
- 점수는 변경 가능성이 있으므로 서비스 계층의 상수/설정 매핑(`score_service`)에서 관리.
